## 一、功能概述

账号注销功能允许用户主动删除其在课程交付CRM系统中的账号和个人信息。本方案采用**软删除+数据脱敏**的策略，在满足《个人信息保护法》（PIPL）和GDPR等法规要求的同时，保持系统数据完整性和业务连续性。

### 核心特性

本方案实现了用户友好的账号注销流程，包含30天缓冲期机制。用户发起注销后，账号立即进入"注销中"状态，禁止登录和使用系统功能。在30天缓冲期内，用户可以通过重新登录恢复账号。缓冲期结束后，系统自动对用户个人信息进行脱敏处理，将手机号、邮箱、真实姓名等敏感信息替换为匿名标识，同时保留用户ID和业务关联数据，确保历史订单、排课记录等业务数据的完整性。

### 法规合规性

本方案符合《中华人民共和国个人信息保护法》第四十七条关于"删除权"的规定，以及GDPR第17条关于"被遗忘权"的要求。用户注销后无法再访问其个人信息，系统不再处理用户的个人数据，满足法规对数据删除的核心要求。同时，通过保留匿名化的业务数据，系统能够满足审计、财务合规和业务分析的需求。

## 二、业务流程设计

### 2.1 用户注销流程

用户发起注销的完整流程分为四个阶段。首先，用户在APP内进入"设置"页面，点击"账号与安全"选项，然后选择"注销账号"功能。系统弹出确认对话框，明确告知用户注销后果，包括删除所有个人信息、订单记录、会员权益等，且30天后无法恢复。用户需要输入手机号和验证码进行身份验证，并勾选"我已了解注销后果"确认框，最后点击"确认注销"按钮。

系统收到注销请求后，立即执行以下操作。首先验证用户身份（手机号+验证码），然后检查用户是否有特殊角色（如老师、销售）和未完成的业务（如未完成的排课、未结算的订单）。如果存在阻塞条件，系统拒绝注销并提示用户先处理相关事项。验证通过后，系统将用户状态标记为"注销中"（`isDeleted = 1`），记录注销时间（`deletedAt = 当前时间`），并立即禁止该账号登录。

### 2.2 账号恢复流程

在30天缓冲期内，用户可以通过重新登录恢复账号。当用户尝试登录时，系统检测到账号处于"注销中"状态且未超过30天，自动弹出恢复确认对话框，询问用户是否恢复账号。用户确认后，系统将`isDeleted`字段重置为0，清空`deletedAt`字段，恢复账号的所有功能和权限。整个恢复过程记录在审计日志中，包括恢复时间、IP地址等信息。

### 2.3 自动脱敏流程

系统通过定时任务（每天凌晨2点执行）自动处理超过30天缓冲期的注销账号。定时任务首先查询所有`isDeleted = 1`且`deletedAt`时间超过30天的用户记录，然后对每个用户执行数据脱敏操作。脱敏操作包括将手机号替换为`DELETED_<userId>`，邮箱替换为`deleted_<userId>@deleted.local`，真实姓名替换为`已注销用户<userId>`，头像URL清空，密码哈希清空，微信号清空等。脱敏完成后，系统将`isDeleted`字段更新为2（表示已脱敏），并记录脱敏时间（`anonymizedAt`）。所有操作记录在审计日志表中，包括原始数据的哈希值（用于审计验证）。

## 三、数据库设计

### 3.1 字段设计

为支持账号注销功能，需要在`users`表中新增以下字段：

| 字段名 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `isDeleted` | `TINYINT(1)` | `0` | 注销状态：0=正常，1=注销中（30天缓冲期），2=已脱敏 |
| `deletionReason` | `TEXT` | `NULL` | 用户填写的注销原因（可选） |
| `anonymizedAt` | `DATETIME` | `NULL` | 数据脱敏完成时间 |

现有的`deletedAt`字段继续保留，用于记录用户发起注销的时间。

### 3.2 数据库修改SQL脚本

```sql
-- ========================================
-- 账号注销功能数据库修改脚本
-- 版本: v1.0
-- 日期: 2026-02-26
-- ========================================

-- 1. 添加注销状态字段
ALTER TABLE users 
ADD COLUMN isDeleted TINYINT(1) NOT NULL DEFAULT 0 
COMMENT '注销状态：0=正常，1=注销中（30天缓冲期），2=已脱敏'
AFTER deletedAt;

-- 2. 添加注销原因字段
ALTER TABLE users 
ADD COLUMN deletionReason TEXT NULL 
COMMENT '用户填写的注销原因（可选）'
AFTER isDeleted;

-- 3. 添加脱敏时间字段
ALTER TABLE users 
ADD COLUMN anonymizedAt DATETIME NULL 
COMMENT '数据脱敏完成时间'
AFTER deletionReason;

-- 4. 添加索引优化查询性能
CREATE INDEX idx_isDeleted_deletedAt ON users(isDeleted, deletedAt);

-- 5. 创建审计日志表（如果不存在）
CREATE TABLE IF NOT EXISTS user_deletion_logs (
  id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  userId INT NOT NULL COMMENT '用户ID',
  action VARCHAR(50) NOT NULL COMMENT '操作类型：request_deletion, restore_account, anonymize_data',
  operatorId INT NULL COMMENT '操作人ID（系统操作为NULL）',
  operatorIp VARCHAR(50) NULL COMMENT '操作人IP地址',
  reason TEXT NULL COMMENT '操作原因或备注',
  dataHash VARCHAR(64) NULL COMMENT '原始数据哈希值（用于审计验证）',
  createdAt DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '操作时间',
  INDEX idx_userId (userId),
  INDEX idx_action (action),
  INDEX idx_createdAt (createdAt)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='用户注销审计日志表';

-- 6. 验证修改结果
SELECT 
  COLUMN_NAME, 
  COLUMN_TYPE, 
  IS_NULLABLE, 
  COLUMN_DEFAULT, 
  COLUMN_COMMENT 
FROM INFORMATION_SCHEMA.COLUMNS 
WHERE TABLE_SCHEMA = DATABASE() 
  AND TABLE_NAME = 'users' 
  AND COLUMN_NAME IN ('isDeleted', 'deletionReason', 'anonymizedAt');
```

### 3.3 数据脱敏规则

当用户账号超过30天缓冲期后，系统自动执行以下脱敏操作：

| 字段名 | 原始值示例 | 脱敏后值 | 脱敏规则 |
|--------|-----------|---------|---------|
| `phone` | `13800138000` | `DELETED_15362121` | 替换为`DELETED_<userId>` |
| `email` | `user@example.com` | `deleted_15362121@deleted.local` | 替换为`deleted_<userId>@deleted.local` |
| `name` | `张三` | `已注销用户15362121` | 替换为`已注销用户<userId>` |
| `nickname` | `小张` | `NULL` | 清空 |
| `avatarUrl` | `https://...` | `NULL` | 清空 |
| `password` | `$2b$10$...` | `NULL` | 清空密码哈希 |
| `wechat` | `wx123456` | `NULL` | 清空 |
| `loginMethod` | `google` | `deleted` | 标记为已删除 |

**保留字段**（不脱敏）：
- `id`：用户ID（主键，用于关联业务数据）
- `roles`：用户角色（用于业务数据查询）
- `createdAt`：注册时间（用于审计）
- `isDeleted`：注销状态（标记为2=已脱敏）
- `deletedAt`：注销时间（用于审计）
- `anonymizedAt`：脱敏时间（用于审计）

## 五、定时任务设计

### 5.1 自动脱敏任务

**任务名称**: `anonymizeDeletedUsers`  
**执行频率**: 每天凌晨2:00执行  
**执行逻辑**:

系统定时任务首先查询所有满足脱敏条件的用户记录，即`isDeleted = 1`且`deletedAt`时间距今超过30天的用户。对于每个符合条件的用户，任务执行以下操作：首先计算原始数据的哈希值并保存到审计日志，然后按照脱敏规则更新用户的个人信息字段，将`isDeleted`状态更新为2（已脱敏），记录`anonymizedAt`时间戳，最后在审计日志表中记录完整的脱敏操作信息。

任务执行完成后，系统生成执行报告，包括处理的用户数量、成功数量、失败数量以及失败原因。报告通过系统通知发送给管理员，确保数据脱敏过程的透明性和可追溯性。

### 5.2 任务实现代码示例

```javascript
// server/scheduledTasks/anonymizeDeletedUsers.ts
import { getDb } from '../db';
import { users, userDeletionLogs } from '../../drizzle/schema';
import { and, eq, lte } from 'drizzle-orm';
import crypto from 'crypto';

export async function anonymizeDeletedUsers() {
  const db = await getDb();
  if (!db) {
    console.error('[定时任务] 数据库连接失败');
    return;
  }

  const thirtyDaysAgo = new Date();
  thirtyDaysAgo.setDate(thirtyDaysAgo.getDate() - 30);

  console.log(`[定时任务] 开始执行用户数据脱敏任务 (${new Date().toISOString()})`);
  console.log(`[定时任务] 查询条件: isDeleted=1 且 deletedAt <= ${thirtyDaysAgo.toISOString()}`);

  // 1. 查找需要脱敏的用户
  const usersToAnonymize = await db.select()
    .from(users)
    .where(
      and(
        eq(users.isDeleted, 1),
        lte(users.deletedAt, thirtyDaysAgo)
      )
    );

  console.log(`[定时任务] 找到 ${usersToAnonymize.length} 个需要脱敏的用户`);

  let successCount = 0;
  let failCount = 0;

  // 2. 逐个处理用户脱敏
  for (const user of usersToAnonymize) {
    try {
      // 2.1 计算原始数据哈希（用于审计）
      const originalDataHash = crypto
        .createHash('sha256')
        .update(JSON.stringify({
          id: user.id,
          phone: user.phone,
          email: user.email,
          name: user.name,
        }))
        .digest('hex');

      // 2.2 执行数据脱敏
      await db.update(users)
        .set({
          phone: `DELETED_${user.id}`,
          email: `deleted_${user.id}@deleted.local`,
          name: `已注销用户${user.id}`,
          nickname: null,
          avatarUrl: null,
          password: null,
          wechat: null,
          loginMethod: 'deleted',
          isDeleted: 2, // 标记为已脱敏
          anonymizedAt: new Date(),
        })
        .where(eq(users.id, user.id));

      // 2.3 记录审计日志
      await db.insert(userDeletionLogs).values({
        userId: user.id,
        action: 'anonymize_data',
        operatorId: null, // 系统自动操作
        operatorIp: null,
        reason: '30天缓冲期结束，自动执行数据脱敏',
        dataHash: originalDataHash,
      });

      console.log(`[定时任务] ✅ 用户 ${user.id} 数据脱敏成功`);
      successCount++;
    } catch (error) {
      console.error(`[定时任务] ❌ 用户 ${user.id} 数据脱敏失败:`, error);
      failCount++;
    }
  }

  console.log(`[定时任务] 数据脱敏任务完成`);
  console.log(`[定时任务] 成功: ${successCount}, 失败: ${failCount}`);

  // 3. 发送执行报告给管理员（可选）
  if (usersToAnonymize.length > 0) {
    // TODO: 调用通知接口发送报告
  }
}
```

## 六、前端开发指南

### 6.1 注销流程页面

前端APP需要实现账号注销的完整用户界面，包括注销入口、确认对话框、验证码输入和结果反馈。注销入口位于"设置"页面的"账号与安全"选项中，点击"注销账号"后弹出确认对话框。

确认对话框需要清晰展示注销后果，包括以下关键信息：删除所有个人信息（手机号、邮箱、姓名等），删除所有订单记录和会员权益，30天内可通过登录恢复账号，30天后永久删除且无法恢复。用户需要完成三个确认步骤：输入手机号进行身份验证，输入短信验证码，勾选"我已了解注销后果"确认框。

### 6.2 账号恢复流程

当用户尝试登录处于"注销中"状态的账号时，系统自动检测并弹出恢复确认对话框。对话框提示用户"您的账号正在注销中，还有X天可恢复。是否恢复账号？"用户点击"恢复账号"按钮后，系统调用`auth.restoreAccount`接口，验证通过后自动恢复账号并跳转到首页。

### 6.3 UI/UX设计建议

注销功能的UI设计应当遵循"安全第一、用户友好"的原则。注销按钮应当使用警示色（如红色），并放置在设置页面的底部位置，避免用户误触。确认对话框需要使用大字号清晰展示注销后果，关键信息使用加粗或高亮显示。验证码输入框应当支持自动填充功能，提升用户体验。恢复账号的入口应当在登录页面明显位置提示，避免用户找不到恢复方式。

## 七、风险控制与安全措施

### 7.1 防止误操作

为防止用户误操作导致账号注销，系统设计了多重确认机制。首先，用户必须手动输入手机号和验证码进行身份验证，避免他人冒用。其次，用户必须勾选"我已了解注销后果"确认框，确保用户充分理解注销的影响。最后，系统提供30天缓冲期，即使用户误操作也有充足的时间恢复账号。

### 7.2 特殊角色限制

对于具有特殊角色的用户（如老师、销售、财务、管理员），系统在注销前进行额外检查。如果用户是老师且有未完成的排课记录，系统拒绝注销并提示用户先完成或转移排课。如果用户是销售且有未结算的订单，系统同样拒绝注销。管理员账号的注销需要超级管理员审批，防止误操作导致系统管理权限丢失。

### 7.3 审计与合规

所有账号注销相关的操作都记录在审计日志表中，包括注销申请、账号恢复、数据脱敏等。审计日志包含操作时间、操作人IP地址、原始数据哈希值等关键信息，确保操作的可追溯性。系统管理员可以随时查询审计日志，验证数据脱敏的正确性和合规性。

## 八、实施计划

### 8.1 开发阶段

账号注销功能的开发分为四个阶段。第一阶段（1-2天）进行数据库准备，执行SQL脚本添加新字段和索引，创建审计日志表。第二阶段（3-5天）进行后端接口开发，实现注销申请、账号恢复、状态查询等API接口，开发定时任务脚本。第三阶段（3-5天）进行前端开发，实现注销流程页面、确认对话框、验证码输入等UI组件。第四阶段（2-3天）进行测试与上线，包括单元测试、集成测试、灰度发布和监控日志。

### 8.2 测试要点

测试阶段需要覆盖以下关键场景：正常注销流程（包括验证码验证、状态更新、登录禁止），账号恢复流程（包括30天内恢复、超过30天无法恢复），数据脱敏任务（包括定时任务执行、脱敏规则验证、审计日志记录），特殊角色限制（包括老师有未完成排课、销售有未结算订单），并发场景（包括同时注销和恢复、定时任务执行期间的操作）。

### 8.3 上线与监控

功能上线后，需要建立完善的监控机制。监控指标包括每日注销申请数量、账号恢复数量、数据脱敏任务执行情况、API接口错误率等。系统管理员应当定期查看审计日志，确保数据脱敏的正确性。如果发现异常情况（如注销数量突然增加、脱敏任务失败等），应当及时排查原因并采取措施。

## 九、常见问题解答

### Q1: 为什么采用软删除而不是真删除？

软删除+数据脱敏的方案能够在满足法规要求的同时，保持系统数据完整性。真删除会导致历史订单、排课记录等业务数据的外键关联失效，影响财务审计和业务分析。软删除方案通过脱敏个人信息，确保用户无法再访问其数据，同时保留匿名化的业务数据用于系统运营。

### Q2: 脱敏后的数据是否可以恢复？

脱敏后的数据无法恢复。系统在脱敏前会计算原始数据的哈希值并保存到审计日志，但不保存原始数据本身。这确保了数据删除的不可逆性，符合法规对"删除权"的要求。

### Q3: 30天缓冲期的法律依据是什么？

30天缓冲期是业界通行做法，平衡了用户体验和法规要求。《个人信息保护法》要求企业在用户请求删除后"及时"删除数据，但没有明确规定具体时限。30天缓冲期给予用户充分的反悔时间，同时不违反"及时删除"的原则。

### Q4: 如果用户是老师或销售，注销会影响历史数据吗？

不会。系统在脱敏用户个人信息后，保留用户ID和角色信息。历史订单和排课记录中的老师ID、销售ID仍然有效，只是显示为"已注销用户<userId>"。这确保了业务数据的完整性和可追溯性。

### Q5: 定时任务失败怎么办？

定时任务包含完善的错误处理和重试机制。如果脱敏操作失败，系统会记录错误日志并在下次执行时重试。管理员可以通过系统通知及时了解任务执行情况，并手动介入处理异常情况。
