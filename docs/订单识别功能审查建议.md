## 🔍 核心问题分析

### 问题1：未集成现有业务规则和数据

**严重程度**: 🔴 高

**问题描述**:

当前LLM提示词是通用的订单信息提取规则，未集成CRM系统已有的业务规则和数据。根据知识库，CRM系统已经积累了大量业务规则：

1. **销售人员和老师名单**: 系统维护了销售管理和老师管理名单，应该作为LLM识别的参考数据
2. **城市管理数据**: 系统有城市管理功能，应该用于城市识别和验证
3. **客户与老师区分规则**: 老师名字不能出现在客户名字段中
4. **合伙人费用计算规则**: 不同城市有不同的合伙人费率（武汉40%、天津50%、其他30%）
5. **理论课老师费用规则**: 理论课老师费用默认为0，除非特别标注
6. **渠道订单号格式验证**: 支付宝28位、微信32位
7. **流量来源别名映射**: 如"瀛姬沐沐(11-20点)" → "瀛姬沐沐"
8. **销售人员别名**: 如"妖渊" → "妍渊"

**影响**:

- 识别准确率无法达到最优
- 无法利用现有数据提高识别质量
- 用户需要手动修正大量可以自动处理的错误

**建议方案**:

修改`recognizeOrderInfo`函数，在调用LLM之前先获取业务数据：

```typescript
1. **设备微信号**：发送消息的微信账号（如"瀛姬喵喵11:00-20:00"），通常不在订单文本中，留空即可

3. **日期**：格式为 MM.DD 或 M月DD日（如"2.18"、"12.17"、"2月19日"）
4. **时间**：格式为 HH:MM-HH:MM 或 HH:MM~HH:MM（如"20:30-21:30"、"13:00~15:30"）
5. **课程名称**：课程内容描述（如"sp课"、"裸足丝袜+埃及艳后"、"基础疯三➕埃及➕tk"、"屈辱剧本一小时"）
6. **老师名称**：必须从已知老师名单中匹配，格式为"XX上"或"XX老师"
7. **城市信息**：必须从已知城市列表中匹配，去掉括号和"单"字
8. **支付金额**：
   - 全款格式："全款XXXX已付"（如"全款1500已付"）
   - 定金+尾款格式："XXXX定金已付 XXXX尾款已付"或"XXXX定金已付 XXXX尾款未付"
   - 客户余额抵扣：提取"余额XXXX-XXXX=XXXX"中的金额
9. **客户名**：可选，通常在金额信息前，**不能**是已知老师名单中的名字
10. **给老师的金额**：格式为"给老师XXX"（如"给老师800"、"给老师750"）
11. **车费**：格式为"报销老师XX车费"或"XX车费"（如"报销老师100车费"、"400车费"）

    - 支付宝：28位数字
    - 微信：32位数字

**提取规则：**
- 如果某个字段在文本中不存在，设置为 null
- 销售人员和老师名称必须从提供的名单中匹配，不匹配则设置为 null
- 城市信息必须从提供的城市列表中匹配，不匹配则设置为 null

- 支付状态根据"已付"或"未付"判断
- 置信度根据提取到的字段完整性和匹配度评估（0-1之间）
- 如果识别到"作废"关键词，设置 isVoided: true

    const userPrompt = `请从以下订单文本中提取信息：

    const response = await invokeLLM({
      messages: [

    const result = JSON.parse(content);

    if (result.salesPerson && trafficSourceAliases[result.salesPerson]) {
      result.salesPerson = trafficSourceAliases[result.salesPerson];
    }
    
    // 3.2 验证老师名不在客户名中
    if (result.customerName && teachers.includes(result.customerName)) {
      console.warn('[OrderRecognition] Customer name matches teacher name, clearing customer name');
      result.customerName = null;
      result.confidence = Math.max(0, result.confidence - 0.1); // 降低置信度
    }
    
    // 3.3 验证渠道订单号格式
    if (result.transactionId) {
      const length = result.transactionId.length;
      if (length === 28) {
        result.paymentChannel = '支付宝';
      } else if (length === 32) {
        result.paymentChannel = '微信';
      }
    }
    
    // 3.4 理论课老师费用规则
    if (result.courseName && result.courseName.includes('理论') && !result.teacherPayment) {
      result.teacherPayment = 0;
    }
    
    // 添加原始文本
    result.originalText = text;
    
    // 确保置信度在 0-1 之间
    if (typeof result.confidence !== 'number' || result.confidence < 0 || result.confidence > 1) {
      result.confidence = 0.7; // 默认置信度
    }

    console.log('[OrderRecognition] Recognition successful, confidence:', result.confidence);

    return result as OrderRecognitionResult;
  } catch (error: any) {
    console.error('[OrderRecognition] Error:', error.message);
    
    // 返回失败结果
    return {
      originalText: text,
      confidence: 0,
    };
  }
}
```

### 问题2：缺少智能学习和反馈机制

**严重程度**: 🟡 中

**问题描述**:

当前方案是"一次性识别"，没有学习机制。用户修正识别错误后，系统无法从中学习，下次遇到类似文本仍会犯同样的错误。

根据知识库，CRM系统应该实现：

1. **错误反馈机制**: 用户可以标记识别错误的字段
2. **智能学习机制**: 系统自动记录用户修正的内容，建立别名映射
3. **持续优化**: 使用修正数据优化LLM提示词

**影响**:

- 识别准确率无法随使用时间提升
- 用户需要重复修正相同的错误
- 无法积累业务知识

**建议方案**:

#### 方案A：添加用户反馈接口

在订单确认页面添加"标记错误"功能，记录用户修正：

```typescript
// 新增API：记录识别错误反馈
recognitionFeedback: protectedProcedure
  .input(
    z.object({
      originalText: z.string(),
      llmResult: z.record(z.any()), // LLM识别结果
      userCorrected: z.record(z.any()), // 用户修正后的结果
      errorFields: z.array(z.string()), // 用户标记的错误字段
    })
  )
  .mutation(async ({ input }) => {
    // 保存到反馈表
    await db.insert(recognitionFeedback).values({
      originalText: input.originalText,
      llmResult: JSON.stringify(input.llmResult),
      userCorrected: JSON.stringify(input.userCorrected),
      errorFields: input.errorFields.join(','),
      createdAt: new Date(),
    });
    
    // 自动分析反馈，更新别名映射
    await analyzeAndUpdateAliases(input);
    
    return { success: true };
  }),
```

#### 方案B：自动学习别名映射

分析用户修正数据，自动建立别名映射：

```typescript
async function analyzeAndUpdateAliases(feedback: FeedbackData) {
  // 分析销售人员名字修正
  if (feedback.errorFields.includes('salesPerson')) {
    const llmName = feedback.llmResult.salesPerson;
    const correctName = feedback.userCorrected.salesPerson;
    
    // 如果同一个错误出现3次以上，自动添加别名映射
    const errorCount = await db.select({ count: sql`count(*)` })
      .from(recognitionFeedback)
      .where(
        and(
          sql`json_extract(llm_result, '$.salesPerson') = ${llmName}`,
          sql`json_extract(user_corrected, '$.salesPerson') = ${correctName}`
        )
      );
    
    if (errorCount[0].count >= 3) {
      await db.insert(aliases).values({
        type: 'salesperson',
        alias: llmName,
        realName: correctName,
      }).onConflictDoNothing();
      
      console.log(`[Learning] Auto-added alias: ${llmName} → ${correctName}`);
    }
  }
  
  // 类似逻辑处理老师名、城市等字段
}
```

### 问题3：未处理作废订单

**严重程度**: 🟡 中

**问题描述**:

根据知识库，系统需要识别"作废"订单，并删除数据库中对应的订单记录。当前方案未实现此功能。

**影响**:

- 作废订单会被错误创建到系统中
- 需要手动删除作废订单
- 数据准确性降低

**建议方案**:

在`OrderRecognitionResult`接口中添加`isVoided`字段：

```typescript
export interface OrderRecognitionResult {
  // ... 其他字段 ...
  
  /** 是否作废订单 */
  isVoided?: boolean;
  
  // ... 其他字段 ...
}
```

在LLM提示词中添加作废识别规则：

```typescript
const systemPrompt = `...

**特殊处理：**
- 如果文本中包含"作废"关键词，设置 isVoided: true
- 作废订单不需要提取详细信息，只需要提取交易编号用于删除

...`;
```

在前端App中处理作废订单：

```typescript
// 前端App订单确认页面
const handleRecognitionResult = async (result: OrderRecognitionResult) => {
  if (result.isVoided) {

              await trpc.orders.deleteByChannelOrderNo.mutate({
                channelOrderNo: result.transactionId,
              });
              Alert.alert('成功', '作废订单已删除');
            }
          },
        },
      ]
    );
  } else {
    // 正常流程：跳转到订单确认页面
    navigation.navigate('OrderConfirm', { orderData: result });
  }
};
```

### 问题4：缺少支付渠道自动识别

**严重程度**: 🟢 低

**问题描述**:

当前方案未根据渠道订单号自动识别支付渠道（微信/支付宝）。根据知识库，支付宝订单号28位，微信订单号32位，可以自动识别。

**影响**:

- 用户需要手动选择支付渠道
- 数据录入效率降低

**建议方案**:

在`OrderRecognitionResult`接口中添加`paymentChannel`字段：

```typescript
export interface OrderRecognitionResult {
  // ... 其他字段 ...
  
  /** 支付渠道（微信/支付宝/付账柜） */

  // ... 其他字段 ...
}
```

在后处理逻辑中添加支付渠道识别：

```typescript
// 3.3 验证渠道订单号格式并识别支付渠道
if (result.transactionId) {
  const length = result.transactionId.length;
  if (length === 28) {
    result.paymentChannel = '支付宝';
  } else if (length === 32) {

  }
```

### 问题5：原始文本未保存到备注字段

**严重程度**: 🟡 中

**问题描述**:

根据知识库，所有通过智能识别导入的订单，原始文本必须保存到"备注"字段，用于数据验证和后续优化。当前方案虽然在`OrderRecognitionResult`中包含`originalText`，但未明确说明要保存到订单的备注字段。

**影响**:

- 无法追溯原始订单文本
- 无法重新解析历史数据
- 无法验证识别准确性

**建议方案**:

在API文档中明确说明，前端App创建订单时，必须将`originalText`保存到订单的`notes`字段：

```typescript
// 前端App创建订单
const createOrder = async (recognitionResult: OrderRecognitionResult) => {

  };
  
  const result = await trpc.orders.create.mutate(orderData);
  return result;
};
```

## 🚀 性能优化建议

### 建议1：使用更快的LLM模型

**当前方案**: 文档中推荐使用GPT-4

```typescript
import { createHash } from 'crypto';

export async function recognizeOrderInfo(text: string): Promise<OrderRecognitionResult> {
  try {
    // 1. 生成文本哈希作为缓存键

    const result = await performRecognition(text);
    
    // 4. 保存到缓存（24小时过期）

    return result;
  } catch (error: any) {
    console.error('[OrderRecognition] Error:', error.message);
    return {
      originalText: text,
      confidence: 0,
    };
  }
}
```

```typescript
const rateLimitMiddleware = t.middleware(async ({ ctx, next }) => {
  const userId = ctx.user?.id;
  if (!userId) {
    throw new TRPCError({ code: 'UNAUTHORIZED' });

      code: 'TOO_MANY_REQUESTS',

    });

  return next();
});
```

### 建议2：敏感信息脱敏

**问题**: 订单文本可能包含客户隐私信息，发送到LLM服务存在泄露风险

**优化方案**:

在发送到LLM之前，对敏感信息进行脱敏处理：

```typescript
function maskSensitiveInfo(text: string): { maskedText: string; masks: Record<string, string> } {
  const masks: Record<string, string> = {};
  let maskedText = text;
  
  // 1. 脱敏手机号
  const phoneRegex = /1[3-9]\d{9}/g;
  maskedText = maskedText.replace(phoneRegex, (match) => {
    const mask = `PHONE_${Object.keys(masks).length}`;
    masks[mask] = match;
    return mask;
  });
  
  // 2. 脱敏身份证号
  const idCardRegex = /\d{17}[\dXx]/g;
  maskedText = maskedText.replace(idCardRegex, (match) => {
    const mask = `IDCARD_${Object.keys(masks).length}`;
    masks[mask] = match;
    return mask;
  });
  
  // 3. 脱敏银行卡号
  const bankCardRegex = /\d{16,19}/g;
  maskedText = maskedText.replace(bankCardRegex, (match) => {
    const mask = `BANKCARD_${Object.keys(masks).length}`;
    masks[mask] = match;
    return mask;
  });
  
  return { maskedText, masks };
}

function unmaskSensitiveInfo(result: any, masks: Record<string, string>): any {
  let jsonStr = JSON.stringify(result);
  
  // 还原脱敏信息
  for (const [mask, original] of Object.entries(masks)) {
    jsonStr = jsonStr.replace(new RegExp(mask, 'g'), original);
  }
  
  return JSON.parse(jsonStr);
}

// 在recognizeOrderInfo中使用
export async function recognizeOrderInfo(text: string): Promise<OrderRecognitionResult> {
  try {
    // 1. 脱敏处理
    const { maskedText, masks } = maskSensitiveInfo(text);
    
    // 2. 使用脱敏后的文本调用LLM
    const response = await invokeLLM({
      messages: [
        { role: "system", content: systemPrompt },
        { role: "user", content: `请从以下订单文本中提取信息：\n\n${maskedText}` },
      ],
      response_format: { type: "json_object" },
    });
    
    const content = response.choices[0].message.content;
    if (!content) {
      throw new Error('LLM returned empty response');
    }

    let result = JSON.parse(content);
    
    // 3. 还原脱敏信息
    result = unmaskSensitiveInfo(result, masks);
    
    // 4. 添加原始文本（未脱敏）
    result.originalText = text;
    
    return result as OrderRecognitionResult;
  } catch (error: any) {
    console.error('[OrderRecognition] Error:', error.message);
    return {
      originalText: text,
      confidence: 0,
    };
  }
}
```

```typescript
async function recordRecognitionMetrics(result: OrderRecognitionResult, userCorrected?: any) {
  const metrics = {

    fieldsExtracted: Object.keys(result).filter(k => result[k] !== null).length,

// 计算准确率
function calculateAccuracy(llmResult: any, userCorrected: any): number {
  const fields = Object.keys(llmResult);
  let correctCount = 0;
  
  for (const field of fields) {
    if (llmResult[field] === userCorrected[field]) {
      correctCount++;
    }
  }
  
  return correctCount / fields.length;
}
```

```typescript
import { trpc } from './lib/trpc';

export function OrderRecognitionScreen() {
```
