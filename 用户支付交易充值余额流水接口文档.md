## 概述

本文档详细描述了课程交付CRM系统中用户支付交易、账户充值、余额查询和流水记录相关的API接口。系统采用tRPC框架构建，所有接口均返回JSON格式数据，支持类型安全的端到端通信。

### 核心功能模块

系统的支付交易模块主要包含以下核心功能：

**账户余额管理**：客户可以通过充值方式向账户中存入资金，形成账户余额。账户余额可用于后续的课程支付，当客户使用余额支付时，系统会自动从账户中扣除相应金额。所有余额变动都会生成详细的流水记录，确保资金流向清晰可追溯。

**流水记录追踪**：系统为每个客户维护完整的账户流水记录，包括充值、消费、退款三种类型。每条流水记录都包含变动前余额、变动后余额、关联订单等详细信息，支持按时间倒序查询，便于客户和管理员核对账目。

**权限控制机制**：系统实施严格的权限控制，普通用户只能查询自己的余额和流水记录，管理员和销售人员可以为客户进行充值操作并查询任意客户的账户信息。所有操作都会记录操作人信息，确保责任可追溯。

### 技术架构说明

系统基于tRPC 11框架构建，采用Superjson进行数据序列化，支持Date、Decimal等复杂类型的自动转换。所有接口都经过全局错误处理中间件处理，确保错误响应统一返回JSON格式，包含详细的错误代码和错误信息。

## 数据模型

### 账户流水表（accountTransactions）

账户流水表记录了客户账户余额的所有变动情况，每次充值、消费或退款都会生成一条新的流水记录。

| 字段名 | 类型 | 必填 | 说明 | 示例值 |
|--------|------|------|------|--------|
| `id` | Integer | 是 | 流水记录唯一标识，自增主键 | `1001` |
| `customerId` | Integer | 是 | 客户ID，关联customers表 | `88` |
| `customerName` | String(100) | 是 | 客户姓名（冗余字段，便于查询） | `张三` |
| `type` | Enum | 是 | 流水类型：`recharge`（充值）、`consume`（消费）、`refund`（退款） | `recharge` |
| `amount` | Decimal(10,2) | 是 | 变动金额，充值和退款为正数，消费为负数 | `500.00` |
| `balanceBefore` | Decimal(10,2) | 是 | 变动前账户余额 | `1000.00` |
| `balanceAfter` | Decimal(10,2) | 是 | 变动后账户余额 | `1500.00` |
| `relatedOrderId` | Integer | 否 | 关联订单ID（消费或退款时填写） | `20260222081554` |
| `relatedOrderNo` | String(50) | 否 | 关联订单号（冗余字段，便于查询） | `20260222081554-000` |
| `notes` | Text | 否 | 备注信息，记录充值原因或消费说明 | `管理员充值` |
| `operatorId` | Integer | 是 | 操作人ID，关联users表 | `1` |
| `operatorName` | String(100) | 否 | 操作人姓名（冗余字段） | `管理员` |
| `createdAt` | Timestamp | 是 | 创建时间，自动生成 | `2026-02-22 10:30:00` |

**索引设计**：

- `customer_idx`：客户ID索引，用于快速查询指定客户的流水记录
- `type_idx`：流水类型索引，用于按类型筛选流水记录
- `order_idx`：关联订单ID索引，用于查询订单相关的流水记录
- `created_at_idx`：创建时间索引，用于按时间范围查询流水记录

**业务规则**：

1. 每次余额变动必须生成流水记录，确保账户变动可追溯
2. `balanceAfter` 必须等于 `balanceBefore + amount`，系统会自动校验
3. 充值和退款的 `amount` 为正数，消费的 `amount` 为负数
4. 流水记录创建后不可修改或删除，保证数据完整性

### 客户表（customers）余额相关字段

客户表中存储了客户的基本信息和账户余额，余额字段与流水表配合使用。

| 字段名 | 类型 | 必填 | 说明 | 示例值 |
|--------|------|------|------|--------|
| `id` | Integer | 是 | 客户ID，自增主键 | `88` |
| `name` | String(100) | 是 | 客户姓名 | `张三` |
| `accountBalance` | Decimal(10,2) | 是 | 账户余额，默认0.00 | `1500.00` |
| `userId` | Integer | 否 | 关联用户ID（App注册用户） | `123` |
| `phone` | String(20) | 否 | 手机号 | `13800138000` |
| `wechatId` | String(100) | 否 | 微信号 | `wechat_user_123` |

**业务规则**：

1. `accountBalance` 字段作为快照字段，实际余额以流水表最新记录为准
2. 系统优先从流水表获取最新余额，如果流水表为空则使用 `accountBalance` 字段
3. 每次余额变动后，系统会同步更新 `accountBalance` 字段

### 1. 查询当前用户余额

**接口路径**：`/api/trpc/account.balance`

**请求方法**：`GET` 或 `POST`

**权限要求**：需要登录（protectedProcedure）

**功能描述**：

查询当前登录用户关联客户的账户余额。系统会首先通过用户ID查找关联的客户记录，然后从账户流水表中获取最新的余额信息。如果流水表中没有记录，则返回客户表中的 `accountBalance` 字段值。该接口主要用于前端App显示用户的当前可用余额。

**请求参数**：无需参数

**请求示例**：

```http
GET /api/trpc/account.balance HTTP/1.1
Host: crm.bdsm.com.cn
Cookie: session=xxx
```

**响应格式**：

```typescript
{
  "result": {
    "data": {
      "json": {
        "success": true,
        "data": {
          "balance": string,        // 当前余额（字符串格式，保留2位小数）
          "customerId": number | null,  // 客户ID
          "customerName": string | null // 客户姓名
        }
      }
    }
  }
}
```

**响应示例**：

```json
{
  "result": {
    "data": {
      "json": {
        "success": true,
        "data": {
          "balance": "1500.00",
          "customerId": 88,
          "customerName": "张三"
        }
      }
    }
  }
}
```

**特殊情况**：

如果当前用户未关联任何客户记录，接口返回：

```json
{
  "result": {
    "data": {
      "json": {
        "success": true,
        "data": {
          "balance": "0.00",
          "customerId": null,
          "customerName": null
        }
      }
    }
  }
}
```

**错误响应**：

```json
{
  "error": {
    "json": {
      "message": "Please login (10001)",
      "code": -32001,
      "data": {
        "code": "UNAUTHORIZED",
        "httpStatus": 401,
        "path": "account.balance"
      }
    }
  }
}
```

## 错误处理

### 错误响应格式

所有API错误都遵循统一的错误响应格式，包含详细的错误代码和错误信息。

```typescript
{
  "error": {
    "json": {
      "message": string,      // 错误消息
      "code": number,         // tRPC错误代码
      "data": {
        "code": string,       // 错误类型代码
        "httpStatus": number, // HTTP状态码
        "path": string,       // 接口路径
        "stack"?: string      // 错误堆栈（仅开发环境）
      }
    }
  }
}
```

### 常见错误代码

| HTTP状态码 | tRPC代码 | 错误类型 | 说明 | 示例场景 |
|-----------|---------|---------|------|---------|
| 400 | -32600 | BAD_REQUEST | 请求参数无效 | 充值金额为负数 |
| 401 | -32001 | UNAUTHORIZED | 未登录或登录已过期 | 未提供有效的Session Cookie |
| 403 | -32003 | FORBIDDEN | 权限不足 | 普通用户尝试为其他客户充值 |
| 404 | -32004 | NOT_FOUND | 资源不存在 | 查询不存在的客户ID |
| 500 | -32603 | INTERNAL_SERVER_ERROR | 服务器内部错误 | 数据库连接失败 |

### 错误处理最佳实践

**前端错误处理建议**：

1. **统一错误拦截**：在tRPC客户端配置全局错误处理器，统一处理所有API错误
2. **用户友好提示**：将技术错误信息转换为用户易懂的提示文案
3. **重试机制**：对于网络错误或服务器错误，实现自动重试机制
4. **日志记录**：记录所有API错误到日志系统，便于问题追踪

**错误处理代码示例**：

```typescript
// tRPC客户端配置
const trpc = createTRPCReact<AppRouter>();

// 错误处理示例
try {
  const result = await trpc.account.recharge.mutate({
    customerId: 88,
    amount: 500.00,
    notes: "客户要求充值"
  });
  
  if (result.success) {
    toast.success(`充值成功！当前余额：${result.data.balanceAfter}`);
  } else {
    toast.error(result.error || "充值失败");
  }
} catch (error) {
  if (error.data?.code === "UNAUTHORIZED") {
    // 跳转到登录页面
    router.push("/login");
  } else if (error.data?.code === "BAD_REQUEST") {
    // 显示参数错误提示
    toast.error(error.message);
  } else {
    // 显示通用错误提示
    toast.error("操作失败，请稍后重试");
  }
}
```

## 业务流程说明

### 充值流程

客户充值的完整业务流程如下：

1. **发起充值请求**：管理员或销售人员在后台管理系统中选择客户，输入充值金额和备注信息，点击"充值"按钮
2. **参数校验**：系统首先验证充值金额是否大于0，客户ID是否有效
3. **查询当前余额**：系统从流水表中查询客户的最新余额，如果流水表为空则从客户表中获取
4. **生成流水记录**：系统创建一条新的流水记录，记录类型为 `recharge`，金额为正数，同时记录变动前后的余额
5. **更新账户余额**：系统更新客户表中的 `accountBalance` 字段，确保与流水表一致
6. **返回充值结果**：系统返回充值前后的余额信息，前端显示充值成功提示

### 消费流程

客户使用账户余额支付订单的完整业务流程如下：

1. **创建订单**：客户在前端App中选择课程，创建订单，选择使用账户余额支付
2. **余额校验**：系统检查客户的账户余额是否足够支付订单金额
3. **扣除余额**：系统从客户账户中扣除订单金额，生成一条类型为 `consume` 的流水记录
4. **关联订单**：流水记录中记录关联的订单ID和订单号，便于后续查询和对账
5. **更新订单状态**：订单状态更新为"已支付"，等待交付
6. **返回支付结果**：系统返回支付成功信息，前端显示订单支付成功

### 退款流程

订单取消或退款的完整业务流程如下：

1. **发起退款请求**：管理员或销售人员在后台管理系统中选择订单，点击"退款"按钮
2. **计算退款金额**：系统根据订单的支付金额计算应退款金额
3. **生成退款流水**：系统创建一条类型为 `refund` 的流水记录，金额为正数，关联原订单
4. **增加账户余额**：系统将退款金额加回客户的账户余额
5. **更新订单状态**：订单状态更新为"已退款"
6. **返回退款结果**：系统返回退款成功信息，前端显示退款成功提示

## 数据一致性保证

### 事务处理

系统在处理充值、消费、退款等涉及余额变动的操作时，采用数据库事务机制确保数据一致性。所有余额变动操作都在同一个事务中完成，包括：

1. 创建流水记录
2. 更新客户账户余额
3. 更新关联订单状态（如果有）

如果任何一步操作失败，整个事务会回滚，确保数据不会出现不一致的情况。

### 余额计算规则

系统采用"流水表优先"的余额计算规则：

1. **查询余额时**：优先从流水表中获取最新的 `balanceAfter` 值，如果流水表为空则使用客户表的 `accountBalance` 字段
2. **更新余额时**：同时更新流水表和客户表，确保两者保持一致
3. **余额校验**：每次变动后，系统会校验 `balanceAfter = balanceBefore + amount`，确保计算正确

### 并发控制

系统采用乐观锁机制处理并发充值或消费的情况：

1. 读取当前余额时，记录版本号或时间戳
2. 更新余额时，检查版本号是否发生变化
3. 如果版本号变化，说明有其他操作已经修改了余额，需要重新读取并重试
4. 最多重试3次，如果仍然失败则返回错误

## 安全性说明

### 认证机制

系统采用Session Cookie认证机制，所有API接口都需要有效的登录Session才能访问。Session通过以下方式获取：

1. **Manus OAuth登录**：前端跳转到Manus OAuth登录页面，登录成功后返回Session Cookie
2. **传统用户名密码登录**：前端调用 `auth.login` 接口，传入用户名和密码，登录成功后返回Session Cookie

Session Cookie的有效期为7天，过期后需要重新登录。

### 权限控制

系统实施严格的权限控制机制：

1. **普通用户**：只能查询自己的余额和流水记录，不能为其他客户充值或查询其他客户信息
2. **销售人员**：可以为自己创建的客户充值，可以查询自己创建的客户的余额和流水记录
3. **管理员**：可以为任意客户充值，可以查询任意客户的余额和流水记录

权限校验在tRPC中间件中实现，所有接口调用前都会进行权限检查。

### 数据加密

系统采用HTTPS协议传输所有API请求和响应，确保数据在传输过程中不被窃取或篡改。敏感信息（如密码）在存储前会进行加密处理。

### 操作审计

系统记录所有充值、消费、退款操作的详细信息，包括：

1. 操作人ID和姓名
2. 操作时间
3. 操作类型和金额
4. 关联的客户和订单

这些审计信息存储在流水表中，便于后续追溯和审计。

## 性能优化建议

### 分页查询

在查询流水记录时，建议使用分页查询，避免一次性加载过多数据。推荐的分页参数：

- 移动端：每页10-20条记录
- Web端：每页20-50条记录

### 缓存策略

对于余额查询等高频接口，建议在前端实施缓存策略：

1. **短期缓存**：余额查询结果缓存5-10秒，减少重复请求
2. **失效机制**：充值、消费、退款操作后立即清除缓存
3. **后台刷新**：定期在后台刷新缓存，确保数据最新

### 批量请求

tRPC支持批量请求功能，可以在一次HTTP请求中调用多个API接口。例如，同时查询余额和流水记录：

```typescript
const [balance, transactions] = await Promise.all([
  trpc.account.balance.query(),
  trpc.account.transactions.query({ limit: 20, offset: 0 })
]);
```

## 常见问题（FAQ）

### 1. 为什么余额查询接口返回的余额与客户表中的不一致？

系统优先从流水表中获取最新余额，如果流水表中有记录，则以流水表的 `balanceAfter` 为准。客户表中的 `accountBalance` 字段仅作为快照字段，可能存在短暂的延迟。

### 2. 充值后余额没有立即更新怎么办？

充值操作是同步的，正常情况下会立即更新余额。如果前端显示的余额没有更新，可能是缓存问题，建议清除缓存或刷新页面。

### 3. 如何查询某个订单相关的流水记录？

可以通过流水记录的 `relatedOrderId` 或 `relatedOrderNo` 字段筛选订单相关的流水记录。系统在流水表上建立了订单ID索引，查询性能较好。

### 4. 流水记录可以修改或删除吗？

不可以。流水记录创建后不可修改或删除，以确保账户变动的完整性和可追溯性。如果发现错误，应通过新的流水记录进行冲正。

### 5. 如何处理充值失败的情况？

如果充值操作失败（例如数据库错误），系统会回滚整个事务，不会生成流水记录，客户余额也不会变化。前端应根据错误响应提示用户重试。

### 6. 系统支持哪些支付方式？

当前系统主要支持账户余额支付。如果需要对接第三方支付（如微信支付、支付宝），需要在订单系统中集成相应的支付接口，充值到账户余额后再使用余额支付。

CREATE TABLE `accountTransactions` (
